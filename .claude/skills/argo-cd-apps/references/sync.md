## 1. `syncPolicy.automated`

Source: `docs/user-guide/auto_sync.md`.

```yaml
spec:
  syncPolicy:
    automated:
      enabled: true      # v3.3+; default true. Pause without dropping the block.
      prune: true        # default false. Delete resources removed from git.
      selfHeal: true     # default false. Re-sync on live drift.
      allowEmpty: false  # default false. Refuse to prune to zero resources.
```

* **No block at all** = manual sync. App goes `OutOfSync` on drift; nothing
  happens until `argocd app sync`.
* **Empty block** `automated: {}` = the most conservative auto-sync â€” no
  prune, no selfHeal, no allowEmpty. Start here.

### `enabled: false` (v3.x)

Added v3.3 (PR #25719 cherry-pick). Explicit kill-switch that keeps the
rest of the block intact â€” useful before `argocd app rollback`, which
requires automated to be off. `null` is treated as enabled. Setting `true`
or removing the field resumes.

```yaml
syncPolicy:
  automated:
    enabled: false   # paused; prune/selfHeal preserved
    prune: true
    selfHeal: true
```

### `selfHeal` interaction with `ignoreDifferences`

Without `RespectIgnoreDifferences=true`, `ignoreDifferences` only changes
diff display. `selfHeal` still slams live state back to git on every
reconcile because the apply still writes the ignored field. Combine
`ignoreDifferences` + `RespectIgnoreDifferences=true` (Â§3) to make
`selfHeal` actually leave a field alone.

### selfHeal is dangerous when

1. **Human is using `kubectl edit` / `kubectl scale`** for emergency
   intervention. selfHeal reverts within `--self-heal-timeout-seconds`
   (default **5s**). The HPA-replicas-during-incident loop is the most
   common bug report. Mitigation: temporarily set
   `argocd.argoproj.io/skip-reconcile: "true"` on the Application, or make
   the field a permanent `ignoreDifferences` + `RespectIgnoreDifferences`.
2. **Another controller owns a field.** SSA's `--force-conflicts` (Argo
   CD's default in SSA mode) takes ownership; with selfHeal, the
   field-ownership war repeats silently. Visible only via `kubectl get -o
   yaml | yq .metadata.managedFields`.
3. **Multi-source apps.** Per `auto_sync.md`: "Disabling self-heal does not
   guarantee that live cluster changes in multi-source applications will
   persist." A change in *another* source can still trigger autosync. For
   true read-only-from-git multi-source, disable `automated` entirely.
4. **Mutating-webhook fields** (Linkerd / Istio sidecars, OPA Gatekeeper
   labels). selfHeal keeps re-firing as the webhook re-mutates. Fix:
   `RespectIgnoreDifferences=true` + `ignoreDifferences`, or Server-Side
   Diff with `IncludeMutationWebhook=true`.

selfHeal cadence is `--self-heal-timeout-seconds` on
`argocd-application-controller`. The
`--self-heal-backoff-cooldown-seconds` flag was deprecated and removed in
v3.3 (commit 8b2e0e1) â€” don't reach for it.

### `prune: true` rule

Never enable `prune` in production until you've committed to a
delete-by-removing-from-git workflow. Argo CD only prunes resources it
believes it owns (matching tracking annotation/label) â€” but typo-blast
radius is real. For high-blast-radius resources, use `Prune=confirm`
per-resource (Â§3).

### `allowEmpty: true`

Default `false`. Without it, an automated sync refuses to prune if the
rendered manifest set is empty â€” guards against a templating bug that
wipes the namespace. Flip to `true` only if "no resources" is a legitimate
desired state (rare).

### ApplicationSet caveat

For Apps generated by an ApplicationSet, editing
`spec.syncPolicy.automated` directly on the Application has **no effect** â€”
the ApplicationSet controller reverts it. Use the ApplicationSet's
`ignoreApplicationDifferences` field
(`docs/operator-manual/applicationset/Controlling-Resource-Modification.md`)
or toggle on the ApplicationSet template.

## 2. `syncPolicy.retry`

```yaml
syncPolicy:
  retry:
    limit: 5            # -1 = unlimited
    backoff:
      duration: 5s      # base; supports "30s", "2m", "1h"
      factor: 2         # exponential multiplier
      maxDuration: 3m   # per-attempt cap
    refresh: false      # --sync-retry-refresh â€” pull a newer commit each retry
```

Defaults `5 / 5s / 2x / 3m` give waits ~5s, ~10s, ~20s, ~40s, ~80s (capped
at 3m). Raise `limit` only when a Sync hook legitimately takes longer than
the controller reconcile interval to settle (e.g. a CRD whose operator
takes 4 minutes); use `-1` for unlimited. `refresh: true` makes retries
pick up *newer* commits â€” useful when a pipeline pushes a fix mid-retry.

**Failure mode:** retries are per-Operation, not per-resource. A retry
replays the *whole* sync, including hooks. An expensive PreSync DB
migration runs on each retry unless scoped with `hook-delete-policy:
BeforeHookCreation` and idempotent migration logic.

## 3. `syncOptions` â€” full table

Source: `docs/user-guide/sync-options.md`. Set at app level
(`spec.syncPolicy.syncOptions: ["Key=Value", ...]`) or per-resource via
`argocd.argoproj.io/sync-options` annotation. **Resource-level wins.**

| Option | Default | What it does | Flip when |
|---|---|---|---|
| `Validate=false` | `Validate=true` | Pass `--validate=false` to apply | CRDs with `RawExtension` (ServiceCatalog); partial / patch-style manifests under SSA â€” required to coexist with `ServerSideApply=true` |
| `CreateNamespace=true` | `false` | Create destination namespace as a precursor (kind ordering puts Namespace first) | This App owns the namespace. Pair with `managedNamespaceMetadata` for labels/annotations. **Gotcha:** if you also commit a `Namespace` manifest, the committed one wins and overrides `managedNamespaceMetadata` |
| `PrunePropagationPolicy=foreground\|background\|orphan` | `foreground` | K8s `DeletionPropagation` for prune | `background` for fast prune of large Deployments; `orphan` to detach without cascading (rare; migrations) |
| `PruneLast=true` | `false` | Run pruning as a final implicit wave after everything is healthy | Moving a Service from old to new Deployment; CRD-deletion order with pending CRs |
| `ApplyOutOfSyncOnly=true` | `false` | Apply only resources that diff `OutOfSync`. **Hooks still run** (unlike selective sync) | Apps with thousands of resources; K8s API pressure. v3.3 fix: cluster-scoped resources now honored (#765); composes with waves (#251) |
| `RespectIgnoreDifferences=true` | `false` | Pre-patches desired state to remove ignored fields from apply, so live value persists | Whenever `ignoreDifferences` is set. Without this, ignoreDifferences only affects the diff display. **Caveat:** only effective when the resource exists; on creation, git wins for the ignored field on first reconcile |
| `ServerSideApply=true` | `false` | `kubectl apply --server-side --force-conflicts` | Manifest > 256 KB (no `last-applied-configuration` annotation); partial-spec patches; adopting an existing resource. **`Replace=true` takes precedence** â€” mutually exclusive |
| `ServerSideDiff=true` | off | (Compare-option, NOT a syncOption.) SSA dry-run, diff predicted live vs actual. Stable since v3.1.0 but **not default** | Want admission webhooks to participate in diff. Set via `argocd.argoproj.io/compare-options` annotation on the Application, or globally via `argocd-cmd-params-cm` `controller.diff.server.side: "true"` |
| `FailOnSharedResource=true` | `false` | Sync **fails** rather than overwriting a resource also managed by another App | Multiple Apps could converge on the same namespace and you want loud failure |
| `Replace=true` | `false` | `kubectl replace` (or create) instead of `apply`. **Destructive** â€” may recreate the resource | Manifest exceeds the 256 KB annotation limit and SSA isn't an option; CRDs whose schema changed unmergeably. Almost always combined with `Force=true`. v3.3 fix #321 brought back from regression |
| `Force=true` | `false` | Delete and recreate. Use with `Replace=true` for re-running Jobs every sync | Resource must be recreated each sync. Per-resource only |
| `SkipDryRunOnMissingResource=true` | `false` | Skip dry-run pre-check when CRD is missing | CR whose CRD is dynamically created in-cluster by another controller (Gatekeeper `ConstraintTemplate` â†’ CRD chain). Once the CRD exists, dry-run runs again |
| `Prune=false` (per-resource) | n/a | Stop *this* resource from being pruned even if removed from git | Pair with `compare-options: IgnoreExtraneous` if you also want App to stay `Synced` |
| `Prune=confirm` (per-resource) | n/a | Sync stays `Syncing` until human approval (UI button, CLI, or `argocd.argoproj.io/deletion-approved: <ISO timestamp>` annotation) | High-blast-radius prune (Namespaces, prod PVCs) |
| `Delete=false` (per-resource) | n/a | Resource survives App cascade deletion | PVCs you want to outlive the App. Distinct from `Prune=false` â€” `Delete` only applies during App deletion |
| `Delete=confirm` (per-resource) | n/a | App deletion stays pending until approved | High-blast-radius cascade (production Namespaces) |
| `ClientSideApplyMigration=true\|false` | `true` (v3.3 transitional) | When `ServerSideApply=true`, migrates field ownership from `kubectl-client-side-apply` â†’ `argocd-controller` before applying | Disable **only** as a v3.3.0/v3.3.1 specific bug remediation. Remove the `=false` override after upgrading past v3.3.2. Custom source manager via `argocd.argoproj.io/client-side-apply-migration-manager: "old-helm-controller"` annotation on the Application |

CLI: `argocd app set NAME --sync-option Key=Val` /
`argocd app unset NAME --sync-option Key`. Per-resource via comma-separated
list in the `argocd.argoproj.io/sync-options` annotation. v3.3 fix #26110 /
#26189 also honors `Replace`/`Force` annotations on the **live** resource
(not just the desired manifest).

## 4. Sync waves and phases

Source: `docs/user-guide/sync-waves.md`. Source-of-truth sort:
`gitops-engine/pkg/sync/sync_tasks.go` lines 14â€“68.

### Phase model

```
PreSync  â†’  Sync  â†’  PostSync                (then SyncFail iff anything failed)
PreDelete â†’  (delete)  â†’  PostDelete         (App cascade-delete only)
```

A phase advances only when *every* resource in the previous phase is
applied **AND healthy** (`sync-waves.md` line 12). This is why your
PostSync hook never runs while one Deployment stays `Progressing`.

```go
// gitops-engine/pkg/sync/sync_tasks.go
var syncPhaseOrder = map[common.SyncPhase]int{
    common.SyncPhasePreSync:  -1,
    common.SyncPhaseSync:     0,
    common.SyncPhasePostSync: 1,
    common.SyncPhaseSyncFail: 2,
}
```

### Wave annotation

```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "5"
```

Default `0`. Negative allowed and runs first. Both resources and hooks
respect waves.

### Sort within a phase

1. **Wave** (lower first)
2. **Kind** â€” Helm-style kind order from `sync_tasks.go`:

   ```
   Namespace, NetworkPolicy, ResourceQuota, LimitRange, PodSecurityPolicy,
   PodDisruptionBudget, ServiceAccount, Secret, SecretList, ConfigMap,
   StorageClass, PersistentVolume, PersistentVolumeClaim,
   CustomResourceDefinition, ClusterRole, ClusterRoleList,
   ClusterRoleBinding, ClusterRoleBindingList, Role, RoleList, RoleBinding,
   RoleBindingList, Service, DaemonSet, Pod, ReplicationController,
   ReplicaSet, Deployment, HorizontalPodAutoscaler, StatefulSet, Job,
   CronJob, IngressClass, Ingress, APIService
   ```
3. **Name** â€” alphabetical tiebreak.

This is **not** "CRDs first" â€” `Namespace` is, `CustomResourceDefinition`
is mid-list. Custom kinds (not in the list) get a non-negative index, so
**custom resources in the same wave as built-ins run after every
built-in**. If a CR depends on a Deployment being up (its controller), put
the controller in wave -1 and the CR in wave 0.

### Inter-wave delay

`ARGOCD_SYNC_WAVE_DELAY` env var on the controller (default **2s**) sleeps
between waves. From the doc: "to give other controllers a chance to react
to the spec change that was just applied. This also prevents Argo CD from
assessing resource health too quickly (against the stale object), causing
hooks to fire prematurely." Bumping to 10s slows multi-wave syncs noticeably
â€” tune only if you have race-prone CRDs.

### v3.3: Namespaces and CRDs land before PreSync

v3.3 changelog #159: "Namespace/CRD creation should happen before PreSync
phase." Pre-3.3 they were applied *in* the Sync phase, breaking PreSync
hooks needing them. Post-3.3, no more hand-rolled `wave: -10` Namespace.

### First-wave selection

`sync-waves.md` line 130: "the first number where any resource is
out-of-sync **or unhealthy**." If a wave-3 Deployment is degraded, the
controller replays from wave 3 â€” skipping waves 0â€“2 even when they're
"fine." Breaks the mental model "waves run once per sync."

### Pruning order is reversed

`sync-waves.md` lines 37â€“39: higher waves prune first. CRDs in wave 0 with
CRs in wave 1 â†’ prune deletes CRs first then attempts CRDs. Good default.

### Pitfall: a Lua health check that always returns Healthy

A custom Lua check that returns `Healthy` unconditionally (or is missing
on a CR whose controller never sets a field) makes waves advance
prematurely. Inverse: a CR with no health check that defaults to
`Progressing` forever blocks waves indefinitely. Either write a real Lua
check (Â§5) or annotate
`argocd.argoproj.io/ignore-healthcheck: "true"`.

## 5. Resource hooks

Source: `docs/user-guide/sync-waves.md`.

### All seven hook types

```yaml
metadata:
  annotations:
    argocd.argoproj.io/hook: PreSync
```

| Type | Runs when |
|---|---|
| `PreSync` | Before the Sync phase. Canonical: DB migration Job |
| `Sync` | In parallel with Sync-phase apply (gets applied with the rest, but lives in `Sync` phase via the annotation) |
| `PostSync` | After every Sync-phase resource is **healthy**. Canonical: smoke-test Job |
| `SyncFail` | **Only** if the sync fails (any phase). Cleanup hooks. "If a SyncFail hook itself fails, Argo CD does not take any additional action" â€” can't recover, only run side effects |
| `Skip` | Tells Argo CD *not* to apply the manifest. Escape hatch for problematic chart resources (canonical: `ingress-nginx-admission-create`) |
| `PreDelete` | App cascade-delete only. Blocks the cascade â€” App stays in `deleting` with `DeletionError` until hook succeeds, is fixed, or is manually deleted |
| `PostDelete` | App cascade-delete only. Runs after children are gone |

Multiple hooks comma-separated: `argocd.argoproj.io/hook: Sync,PreSync`
runs the same resource in both phases.

### All three delete policies

```yaml
metadata:
  annotations:
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
```

| Policy | When the hook resource is deleted |
|---|---|
| `HookSucceeded` | After hook completes successfully |
| `HookFailed` | After hook fails |
| `BeforeHookCreation` | Previous hook deleted just before new one is created. **Default.** Pairs with `metadata.name` (NOT `generateName`) |

Multiple comma-separated: `HookSucceeded,HookFailed,BeforeHookCreation` =
"always delete." v3.3 fix #144: `HookSucceeded` now runs after the hook is
*fully* completed (earlier versions had a delete-while-terminating race).

### Canonical PreSync DB-migration

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate                # not generateName â€” required for BeforeHookCreation
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
    argocd.argoproj.io/sync-wave: "-1"
spec:
  ttlSecondsAfterFinished: 360    # K8s deletes the Job after this; alternative to HookSucceeded
  backoffLimit: 1                 # in-cluster retry; outer sync retry is syncPolicy.retry
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: registry.example.com/payments-migrator:1.42
          envFrom: [{ secretRef: { name: payments-db-creds } }]
          command: ["/bin/sh", "-c", "alembic upgrade head"]
```

Field rationale:

* `metadata.name` not `generateName` â€” `BeforeHookCreation` identifies the
  previous hook by name. With `generateName`, every sync leaves a zombie
  Job behind ("hook leak"); the annotation can't find a previous hook to
  delete.
* `wave: "-1"` â€” runs before any other PreSync hook (warmup,
  secret-rotation at wave 0).
* `backoffLimit: 1` â€” Job-internal retry. The *outer* sync retry is
  `syncPolicy.retry`. Pick which one wins.
* `ttlSecondsAfterFinished` â€” alternative to `HookSucceeded` cleanup. v3.3
  fix #646 ("stuck hook issue when a Job has ttlSecondsAfterFinished") â€”
  be on v3.3.0+.

### Hook namespace must exist

The hook resource's namespace must exist or you must set
`CreateNamespace=true` on the Application. v3.3 #159 made
Namespace/CRD creation happen before PreSync, so this is no longer the
trap it used to be.

### Selective sync skips hooks

`sync-waves.md` line 35: "Hooks do not run during a selective sync
operation." `argocd app sync --resource Deployment/foo` skips the PreSync
DB migration. Either accept that or don't selective-sync (Â§8).

## 6. Health assessment

Source: `docs/operator-manual/health.md`.

### Built-in health checks

Hardcoded in `gitops-engine/pkg/health` for:

```
PersistentVolumeClaim, Pod, Service,
apiregistration.k8s.io/APIService,
apps/{DaemonSet, Deployment, ReplicaSet, StatefulSet},
argoproj.io/Workflow,
autoscaling/HorizontalPodAutoscaler,
batch/Job, batch/CronJob,
extensions/Ingress, networking.k8s.io/Ingress
```

Notable rules:

* Deployment/StatefulSet/DaemonSet/ReplicaSet: `observedGeneration ==
  generation` AND `updatedReplicas == desiredReplicas`. **It's
  `updatedReplicas`, not `readyReplicas`** â€” a stuck rollout reports
  `Progressing`, not `Degraded`.
* Service `LoadBalancer`: `status.loadBalancer.ingress[*]` must have at
  least one `hostname` or `IP`. Ingress: same.
* Job: `spec.suspended: true` â†’ `Suspended` (App also reports `Suspended`).
* PVC: `status.phase == Bound`.

`argoproj.io/Application` has no built-in health check since 1.8 â€” restore
via `health.md` lines 35â€“59 if doing app-of-apps with sync waves.

### Rollup rule

App health = **worst** child resource health, by priority (most â†’ least
healthy): `Healthy â†’ Suspended â†’ Progressing â†’ Missing â†’ Degraded â†’
Unknown`. So one Missing + one Degraded = `Degraded` (Degraded outranks
Missing in worseness).

Health is **not inherited from grandchildren** (`health.md` lines
234â€“254). A Deployment's health uses only its own status fields â€” it
doesn't read its Pod statuses. To make a CR's health depend on its
children, the CR's controller must surface them in `status` and your Lua
check must read them.

### Custom Lua health check

Drop in `argocd-cm`. Key format:
`resource.customizations.health.<group>_<kind>`. Skeleton:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  resource.customizations.health.example.com_MyResource: |
    hs = { status = "Progressing", message = "" }
    if obj.status ~= nil and obj.status.conditions ~= nil then
      for _, c in ipairs(obj.status.conditions) do
        if c.type == "Ready" and c.status == "True" then
          hs.status = "Healthy";  hs.message = c.message;  return hs
        end
        if c.type == "Ready" and c.status == "False" then
          hs.status = "Degraded"; hs.message = c.message;  return hs
        end
      end
    end
    return hs
```

Contract: global `obj` = live resource (unstructured). Return
`{ status, message }`. Valid statuses: `Healthy | Progressing | Degraded |
Suspended` (Missing and Unknown are computed by Argo CD).

A configured check **wins over** the built-in Go check â€” useful when the
built-in HPA / Ingress check is wrong for your stack.

### `useOpenLibs: true` â€” security implication

Lua std libs are **disabled by default**. To enable for a kind:

```yaml
data:
  resource.customizations.useOpenLibs.example.com_MyResource: "true"
```

This grants `string.format`, `os.*`, `io.*`, etc. Implication: a malicious
or buggy custom check could read environment, open files, or hang the
controller â€” there are no sandbox limits beyond Lua's defaults. Don't
enable unless required, and review the Lua source as part of normal
manifest review.

### Per-resource opt-out

```yaml
metadata:
  annotations:
    argocd.argoproj.io/ignore-healthcheck: "true"
```

Resource excluded from the App rollup. Use for known-flaky DaemonSets
that shouldn't gate PostSync hooks.

### v3.3 health additions

Per `docs/operator-manual/upgrading/3.2-3.3.md` lines 117â€“124:

* `ceph.rook.io/CephCluster`, `ceph.rook.io/CephObjectStore`
* `objectbucket.io/ObjectBucketClaim`
* `keda.sh/ScaledJob`
* `services.cloud.sap.com/{ServiceBinding, ServiceInstance}`
* `*.cnrm.cloud.google.com/*` (GCP Config Connector wildcard)
* `grafana-org-operator.kubitus-project.gitlab.io/*`

Plus fixes for HPA, KubeVirt, Crossplane v2, GCP Config Connector. Delete
custom Lua you wrote for these post-upgrade.

## 7. Diffing strategies

Source: `docs/user-guide/diffing.md`, `diff-strategies.md`,
`compare-options.md`.

### The three strategies

| Strategy | Status | How it computes |
|---|---|---|
| **Legacy 3-way merge** | Default for non-SSA apps | live âŠ• desired âŠ• `last-applied-configuration` annotation |
| **Server-Side Diff (SSD)** | **Stable since v3.1.0; not default.** Recommended path | SSA dry-run, compare predicted live to actual live |
| **Structured-merge-diff** | **Discontinued** | Field-ownership via structured-merge-diff library |

Per `diff-strategies.md` line 22: structured-merge-diff is **discontinued**
in favour of Server-Side Diff. For SSA-using apps, **also turn on
Server-Side Diff.**

### Server-Side Diff â€” why and how

Pros:
* Validating webhooks fire at diff time. You see "this manifest is
  invalid" *before* sync.
* With `IncludeMutationWebhook=true`, mutating-webhook defaults are
  visible in the diff (Linkerd injection, etc.) â€” kills the "OutOfSync
  after every successful sync" footgun.

Cons:
* One dry-run apply per resource (cached, re-run on refresh / new revision
  / spec change / live `resourceVersion` change).
* Skipped on resource creation (no live to compare; webhooks won't run for
  non-existent resources).

Enable via `compare-options` annotation on the **Application**:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  annotations:
    argocd.argoproj.io/compare-options: ServerSideDiff=true,IncludeMutationWebhook=true
```

Globally via `argocd-cmd-params-cm`:

```yaml
data:
  controller.diff.server.side: "true"
```

Per-app disable when globally enabled:
`compare-options: ServerSideDiff=false`. Verify install state:
`kubectl -n argocd get cm argocd-cmd-params-cm -o yaml | grep
diff.server.side`.

### `ignoreDifferences` â€” pointers, JQ, managers

```yaml
spec:
  ignoreDifferences:
    # JSON pointers (RFC 6902) â€” `/` encoded as ~1 inside a key
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas
        - /metadata/labels/some.example.com~1key

    # JQ paths â€” more powerful; supports list-element selection
    - group: apps
      kind: Deployment
      jqPathExpressions:
        - .spec.template.spec.initContainers[] | select(.name == "injected-init")

    # Field-manager-based ignore â€” "any field where this manager wrote â†’ ignore"
    - group: '*'
      kind: '*'
      managedFieldsManagers:
        - kube-controller-manager
        - linkerd
```

`name` and `namespace` can scope a rule. JQ supports list-element selection
that JSON pointers can't. JQ timeout is **1 second** by default â€” bump via
`argocd-cmd-params-cm` `ignore.normalizer.jq.timeout: "5s"`.

**Without `RespectIgnoreDifferences=true`, `ignoreDifferences` only affects
the diff display.** The apply still writes the field, so on every sync the
controller hammers the live value back. Always pair them when you want
"leave the field alone."

Combined "let HPA own replicas" pattern:

```yaml
spec:
  ignoreDifferences:
    - group: '*'
      kind: '*'
      managedFieldsManagers: [kube-controller-manager]
  syncPolicy:
    syncOptions:
      - ServerSideApply=true
      - RespectIgnoreDifferences=true
```

### Other compare-options

* `argocd.argoproj.io/compare-options: IgnoreExtraneous` (per-resource) â€”
  resource may exist without being in git, App stays `Synced`. Often paired
  with `Prune=false`.
* `argocd-cm` cluster-wide:
  ```yaml
  data:
    resource.compareoptions: |
      ignoreResourceStatusField: all       # default â€” ignore status on every resource
      ignoreAggregatedRoles: true          # ClusterRoles using aggregationRule
  ```
* `KnownTypeFields` (`diffing.md` lines 150â€“197) â€” for CRDs embedding K8s
  types where custom marshalling reformats values (`100m` â†’ `0.1`):

  ```yaml
  data:
    resource.customizations.knownTypeFields.argoproj.io_Rollout: |
      - field: spec.template.spec
        type: core/v1/PodSpec
  ```

  Built-in types: `core/Quantity`, `meta/v1/Duration`, plus
  `util/argo/normalizers/diffing_known_types.txt`.

## 8. Selective sync

Source: `docs/user-guide/selective_sync.md`.

```bash
argocd app sync NAME --resource apps:Deployment:foo
argocd app sync NAME --resource ':ConfigMap:bar' \
                     --resource 'networking.k8s.io:Ingress:web'
```

The whole doc, paraphrased: **hooks are not run, the sync is not recorded
in history, rollback is not possible.**

Different from `ApplyOutOfSyncOnly=true`:

| | Selective sync | `ApplyOutOfSyncOnly=true` |
|---|---|---|
| Scope selection | Manual (`--resource`) | Automatic (anything OutOfSync) |
| Hooks | **Skipped** | Run normally |
| History | Not recorded | Normal |
| Use when | Surgical hot-fix of one resource | Large apps; K8s API pressure |

### Other relevant `argocd app sync` flags

```bash
argocd app sync NAME --dry-run                  # planning only; webhooks DO NOT run
argocd app sync NAME --prune                    # prune even on manual
argocd app sync NAME --replace --force          # one-shot Replace+Force
argocd app sync NAME --strategy hook            # default; hooks run
argocd app sync NAME --strategy apply           # vs. hook
argocd app sync NAME --revision <sha>           # pin a revision
argocd app sync NAME --local ./manifests        # apply local manifests
argocd app sync NAME --apply-out-of-sync-only   # one-shot ApplyOutOfSyncOnly
argocd app sync NAME --server-side              # one-shot ServerSideApply
```

`--dry-run` runs the controller's planning logic but **does not** invoke
`kubectl --dry-run=server`. Webhook validation does not run. For
webhook-aware preview, use Server-Side Diff and read the diff output.

## 9. Sync windows

Source: `docs/user-guide/sync_windows.md`. Live on the **AppProject**, not
the Application â€” project-admin tool. App authors should still know how
they affect their App.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
spec:
  syncWindows:
    - kind: deny
      schedule: '0 22 * * *'        # cron: 10pm
      timeZone: 'Europe/Amsterdam'
      duration: 8h
      namespaces: ['prod']
      manualSync: false             # true = humans can override
      syncOverrun: false             # true = let in-progress syncs continue
    - kind: allow
      schedule: '0 9 * * 1-5'       # 9am Monâ€“Fri
      duration: 8h
      applications: ['*-prod']
      manualSync: true
```

Resolution rules (`sync_windows.md` lines 36â€“41):

* No matching window â†’ all syncs allowed.
* Active `allow` matches â†’ allowed (any non-active means blocked).
* Active `deny` matches â†’ blocked.
* Both match â†’ **deny wins.**

`manualSync: true` lets a human bypass `deny`. `syncOverrun: true` lets a
sync that *started* legally continue past the window boundary â€” **all
active deny windows must have it enabled** for an in-progress sync to
continue when transitioning into deny. Most teams want it on for
long-running migrations.

App-author awareness: pipelines on `argocd app wait` may time out in a
windowed project. `argocd app get NAME` shows `SyncWindow: Sync Denied`
and assigned windows.

## 10. Pruning gotchas

### "I deleted from git but it's still there" â€” 7-cause checklist

In order of likelihood:

1. **Auto-prune is off.** `automated.prune` defaults to `false`. Without
   it, removing the manifest â†’ App goes `OutOfSync` but the resource
   lives. Fix: enable, or `argocd app sync --prune`.
2. **`Prune=false` annotation lingering on the resource** from earlier
   intent. Check `kubectl get RESOURCE -o yaml | grep
   argocd.argoproj.io/sync-options`.
3. **Resource not tracked by this App.** Argo CD only prunes resources it
   thinks it owns (matching tracking annotation/label). Created
   out-of-band â†’ no prune. Verify
   `argocd.argoproj.io/tracking-id: APP:GROUP/KIND:NS/NAME` (or label
   `app.kubernetes.io/instance: APP`, depending on tracking strategy).
4. **`allowEmpty: false` and the new manifest set is empty.** Auto-sync
   refuses. Either commit *something* or set `allowEmpty: true`.
5. **Sync window denying.** `argocd app get NAME` reveals.
6. **Resource has its own finalizer** (PVC mounted by a Pod, namespace
   with stuck resources). DELETE goes through; K8s waits.
7. **`PrunePropagationPolicy=orphan`** â€” children released, not
   cascaded.

### Orphan resources (project-level)

`docs/user-guide/orphaned-resources.md`. On `AppProject`:

```yaml
spec:
  orphanedResources:
    warn: true
    ignore:
      - kind: Secret
        name: '*.example.com'
```

Surfaces resources in the project's namespace not managed by any App.
**Doesn't prune** â€” just shows. Disable on busy namespaces (`kube-system`)
â€” perf hit per the doc. Never orphaned: namespaced resources denied by
project, `default` ServiceAccount, `kubernetes` Service in `default`,
`kube-root-ca.crt` ConfigMap.

### Finalizers â€” cascade delete

`docs/user-guide/app_deletion.md`:

```yaml
metadata:
  finalizers:
    - resources-finalizer.argocd.argoproj.io                # foreground (default)
    # - resources-finalizer.argocd.argoproj.io/background   # background
```

Without one, `kubectl delete app NAME` removes the App CR but **leaves all
child resources running**. With it, deletion cascades. `argocd app delete
--cascade` adds the finalizer for you. Foreground = block until children
gone (safer); background = return immediately, GC async (faster; can leave
inconsistent state).

### Namespace deletion ordering

When the App's destination namespace is *also* in the App's manifest set
**and** has the App's tracking annotation, deleting the App will delete
the namespace, which deletes *every* resource in that namespace â€”
including ones owned by *other* Apps that happened to deploy there. Silent
nuclear option.

Mitigations:
* `Delete=confirm` on the Namespace.
* Don't put a Namespace in an App that doesn't own the namespace
  exclusively.
* Don't manually annotate Namespaces with `tracking-id` â€” `sync-options.md`
  line 493: "manually managing the tracking metadata on a namespace is
  strongly discouraged."

### PV/PVC retention

For PVCs that should survive App deletion:

```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-options: Delete=false
```

The PV itself follows the storage class's `reclaimPolicy`: `Delete` â†’ PV
and disk removed when PVC is; `Retain` â†’ disk persists. Argo CD's
`Delete=false` only affects the *Argo cascade*, not K8s reclaim.
