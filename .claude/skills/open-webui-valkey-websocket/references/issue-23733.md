# Issue #23733 — Socket.IO frame amplification (the structural bug)

Status as of 2026-05-10: **OPEN.** Opened 2026-04-14. Last update 2026-05-08. No merge ETA. The maintainer's preferred design (Yjs document streaming) is a significant refactor; multiple operator-contributed PRs have been closed without merge.

This is the single most important thing to understand before running Open WebUI multi-pod. Without grasping it, the perfect env config still won't prevent the deployment from dying at scale. With it, the right knob to turn and the right load tests to run become obvious.

## What the bug is

Open WebUI's chat streaming architecture is **full-state-on-every-emit**. Every time the LLM emits a token via SSE, the backend:

1. Appends the token to the in-flight assistant message.
2. **Re-serializes the entire accumulated assistant message** (now message_length characters long).
3. Emits it as a new Socket.IO frame.

The frame contains the *full message-so-far*, not a delta. If the response ends up being 4000 tokens long, the backend emits 4000 frames, of average length ~2000 characters each. Total emitted bytes ≈ N² / 2 where N is the final response length. The browser receives this fine — it just overwrites `message.content` on every frame and re-renders the Markdown. Single-pod, no `WEBSOCKET_MANAGER`, this is annoying but tolerable.

## Why it goes nuclear with `WEBSOCKET_MANAGER=redis`

When `WEBSOCKET_MANAGER=redis`, Open WebUI uses `socketio.AsyncRedisManager`. Every `sio.emit(...)` call goes through Valkey pub/sub: the frame is serialized and broadcast on a Valkey channel that **every pod is subscribed to**, so the receiving pod can find the right WebSocket session and forward the frame.

So for a single 4000-token response with a 4-pod, 6-node Valkey Sentinel cluster:

- 4000 emits × ~2000 byte average frame = ~8 MB of *useful* data.
- 4000 emits × full serialization round-trip through Valkey pub/sub = **4000 publish operations**.
- Each publish fans out to all subscribed pods.
- Multiply by concurrent streams: 100 simultaneous reasoning-model responses = 400,000 publishes per minute on the Valkey master.

The issue author (`Ithanil`) measured **~10,000× infrastructure-traffic amplification** on a 6-node Valkey × 4-worker × 100-concurrent-stream setup. Their words: *"for users with many active sessions, models with high token/s ultimately kill the system and Redis."*

This is the most likely root cause of "we had to disable multi-pod and websockets in production." It is structural, not a bug in the conventional sense — the architecture is intentionally this way. It just doesn't scale.

## The maintainer's design rationale (verbatim)

tjbck on issue #23733, 2026-04-17 — the most informative single comment in the corpus. Worth reading in full:

> "The current architecture is full-state-on-every-emit by design. Every Socket.IO frame carries the complete rendered content of the assistant message. This makes the frontend entirely stateless with respect to streaming: it receives a string, sets `message.content`, and hands it to the Markdown renderer. If a WebSocket frame is dropped, the connection flaps, the user switches tabs and comes back, or the browser GC causes a missed event, the very next frame self-corrects because it contains the complete truth. There is no accumulated client-side state that can drift out of sync with the backend.
>
> This is genuinely useful. Streaming Markdown is hard — partial code fences, half-rendered tables, escaped characters mid-sequence. With deltas you have to reason about state machines and reconciliation. With full-state-per-frame, you don't. The Markdown renderer is idempotent on the *current* string and you don't care about what came before.
>
> The cost of this reliability is quadratic total bandwidth. For short messages this is fine. For long reasoning model outputs, especially with tool-calling and structured content, the constant gets large enough that it dominates infrastructure costs.
>
> The lowest-risk immediate improvement is increasing the default `delta_chunk_size`. With the current default of 1 it emits on every single token. Raising it to something like 5 or 10 cuts total bandwidth proportionally with zero code changes and zero change to the reliability model. It is still O(N²) but with a meaningfully smaller constant, and for the majority of real-world responses this may bring cumulative traffic into acceptable range without touching the architecture.
>
> The proper fix is moving to a Yjs document where the backend only ever sends CRDT updates and the client maintains the full state through document merges. This handles ordering, missing updates, and reconnection at the protocol level rather than via the workaround of resending everything. It's also a significant architectural change that touches the entire chat rendering pipeline. We're working toward it but no ETA."

## The mitigation — `CHAT_RESPONSE_STREAM_DELTA_CHUNK_SIZE`

```bash
CHAT_RESPONSE_STREAM_DELTA_CHUNK_SIZE=10
```

Default is `1` (emit on every token). Setting to `10` means the backend buffers 10 tokens before emitting, cutting the number of Socket.IO frames per response by 10×. Bandwidth is also cut by ~10× (each frame is slightly larger on average, but there are 10× fewer of them, and the per-frame protocol overhead drops to 1/10). The streaming UX is still smooth — 10 tokens at typical inference rates is ~50–200ms, well within human perception of "live."

Empirical numbers from operator reports on #16219 and #23733:

| Value | Frames per 4000-token response | Cumulative emit bytes (rough) | Subjective UX |
|---|---|---|---|
| 1 (default) | 4000 | 8 MB | Smoothest |
| 5 | 800 | 1.6 MB | Indistinguishable from default |
| 10 | 400 | 800 KB | Indistinguishable from default in most cases |
| 20 | 200 | 400 KB | Slightly chunky on slow models |
| 50 | 80 | 160 KB | Visible chunkiness |

**Recommended starting point: 10.** If Valkey pub/sub bandwidth or Socket.IO frame rate is still the bottleneck, move to 20. Beyond that the UX starts to suffer. The constant remains O(N²), so very long responses (>10K tokens) are still expensive — but most production traffic isn't 10K-token reasoning chains, and `=10` brings the typical chat into acceptable territory.

The setting is read at process start; restart pods after changing.

## Why the existing PRs didn't merge

Operators have proposed three approaches; none have landed:

1. **#23735 — emit deltas instead of full state.** Closed. Maintainer position: breaks the "self-healing client" guarantee. If a delta is dropped, the client desyncs from the server and there's no recovery path without a full refresh.
2. **#23736 — resumable WS streaming with seq-based replay.** Closed. Adds backend state (the replay log in Valkey) that the architecture is explicitly trying to avoid; also doesn't address the core bandwidth issue, only adds robustness around dropped frames.
3. **#24124, #24126, #24171 — Yjs document streaming.** All closed. These are in the right direction architecturally (matches tjbck's stated preference) but not yet polished enough; the chat-rendering pipeline uses these documents in many places and the PRs only retrofit Yjs into the streaming hot path.

The proper fix is therefore non-trivial and unlikely to land soon. Plan around the mitigation, not around an upcoming patch.

## Operational implications

- **Set `CHAT_RESPONSE_STREAM_DELTA_CHUNK_SIZE=10` before doing anything else.** This single env var is the difference between "multi-pod is feasible" and "multi-pod melts."
- **Reasoning models and tool-calling make the problem worse.** They produce longer, more structured assistant messages, which scales the constant. For traffic heavy on `o1`-style or `deepseek-r1`-style models, prefer `=20`.
- **Valkey master CPU is the canary.** Watch it during multi-pod load tests — that single metric tells the whole story. With the default `=1` it pins a CPU core at ~30 concurrent streams; with `=10` it comfortably handles 200+.
- **Per-stream client bandwidth is also cut.** Users on slow connections benefit too — fewer round-trips for the same content.
- **Don't try to "fix" this by disabling `WEBSOCKET_MANAGER=redis`.** Without the Redis manager there's no multi-pod WebSockets — every emit only reaches the pod that owns the session. The amplification is the price of cross-pod fan-out.

## How to verify the mitigation is in effect

After setting the env var and rolling pods:

1. Open browser devtools → Network → WS, then start a long chat.
2. Observe inbound frames. The default emits one frame per token (often >10/sec). With `=10`, frames arrive in bursts of ~50–200ms.
3. On the Valkey master, run `valkey-cli MONITOR` for 30 seconds during a streaming response and grep for `PUBLISH` lines from the Socket.IO channel. Compare counts before/after.
4. Watch Socket.IO server logs (with `WEBSOCKET_SERVER_LOGGING=True`) for emit rate.

## Long-term outlook

If/when the Yjs-document architecture lands, the streaming protocol becomes:

- Backend maintains the assistant message as a Yjs document.
- Each token append is a CRDT update sent over Socket.IO.
- Updates are O(1) per token, idempotent, and self-merge on reconnect.
- Total bandwidth becomes O(N) instead of O(N²).

This eliminates the amplification. Until it lands, the env knob is the only mitigation available.
